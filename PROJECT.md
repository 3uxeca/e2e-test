# 프로젝트: Cartographer — 자율 E2E 테스트 생성 에이전트 (Week 1)

## 배경
나는 에이전트 주도 개발 역량을 실험하기 위해, 기존에 내가 만든 Next.js 웹앱을
대상으로 E2E 테스트를 자율 생성·실행하는 에이전트 시스템을 구축하려고 한다.
이 프로젝트는 별도 디렉토리에서 진행하며, 기존 웹앱 코드는 수정하지 않는다.

## 최종 목표
URL과 간단한 앱 설명만 주면, 에이전트가 자율적으로 E2E 테스트 스위트를
생성·유지보수하는 시스템.

## 주차별 로드맵
- **1주차: 단일 에이전트 + 로컬 + Opus 단일 모델** — baseline 구축, 병목 식별
- **2주차: 멀티 에이전트 + 로컬 + 모델 분리(Opus/Sonnet)** — 1주차 baseline 대비
  멀티 에이전트와 모델 분리의 효과를 동시에 측정
- **3주차: 멀티 에이전트 + 스테이징 + 모델 분리** — 환경 적응력 검증

각 주차의 결과 데이터가 다음 주차 실험 설계의 근거가 된다.
이번 작업은 1주차이므로 단일 에이전트 + 로컬 + Opus 단일 모델에 한정한다.

## 1주차 목표
단일 에이전트(Opus 4.7)가 로그인 + 핵심 기능 1개에 대한 Playwright E2E
테스트를 자율 생성하고 실행할 수 있게 한다. 동시에 2주차 멀티 에이전트 +
모델 분리 도입의 근거가 될 병목 데이터를 수집한다.

## 1주차 핵심 연구 질문
- 단일 에이전트가 통제된 환경에서 의미 있는 E2E 테스트를 자율 생성할 수 있는가?
- 어느 단계에서 토큰/시간 병목이 발생하는가?
- 어떤 역할을 분리하면 2주차 멀티 에이전트 구성이 가장 효과적일까?
- 어느 단계가 Opus의 추론 능력이 꼭 필요하고, 어느 단계는 Sonnet으로 충분할까?

## 기술 스택
- Claude Agent SDK (TypeScript)
- 1주차 모델: **Claude Opus 4.7 단일 사용** (claude-opus-4-7)
- Playwright MCP
- 대상 웹앱: Next.js, 내부 백엔드 API, 웹소켓 사용 (1주차에선 웹소켓 제외)
- 인증: 숨겨진 로그인 URL + 이메일/비밀번호 (환경변수로 주입)

## 환경 (1주차)
- 대상 웹앱은 **로컬**에서 실행 중이다 (환경변수 TARGET_URL로 주입,
  예: http://localhost:3000).
- 테스트 전용 계정으로 로그인한다 (환경변수로 주입).
- 시드 데이터: 사람이 미리 준비한 테스트용 데이터가 DB에 들어가 있음.

## 1주차 스코프 제외
- 실시간 웹소켓 기반 기능은 1주차 테스트 대상에서 제외한다.
  (로컬에서 웹소켓 stream을 수동 실행해야 하는 환경 제약 때문)
- 단계 1(기능 매핑)에서 웹소켓 의존 기능을 발견하면 문서에 기록만 하고,
  단계 2(우선순위 선택)에서는 후보에서 제외해라.
- 사람이 명시적으로 허용하기 전까지 웹소켓 기반 기능에 대한 테스트는
  작성하지 마라.

## 안전 가드레일

### 환경별 차등 적용 (주차별 로드맵 참고)
가드레일은 환경에 따라 다르게 적용된다. 환경 정보는 환경변수 ENV로 주입하고,
오케스트레이터가 ENV 값에 따라 적절한 가드레일 정책을 에이전트에게 주입한다.

| 항목 | 로컬 (1주차) | 스테이징 (3주차) | 프로덕션 (향후) |
|---|---|---|---|
| 관리자 페이지 접근 | 허용 | 허용 | 금지 |
| 비가역 액션 (삭제 등) | 허용 | 허용 | 사람 승인 필요 |
| 결제/외부 알림 발송 | 허용 (mock) | 사람 승인 필요 | 금지 |

### 환경 무관 공통 가드레일 (모든 주차에서 적용)
- 같은 액션 5회 이상 반복 시 중단 (무한 루프 방지)
- 단계당 토큰 사용량 상한 (아래 "토큰 상한" 섹션 참고)
- 어떤 액션이 애매하면 실행하지 말고 사람에게 물어봐라

### 1주차 (로컬) 적용 가드레일 요약
- 관리자 페이지 포함한 모든 기능 영역 탐색·테스트 가능
- 비가역 액션도 자유롭게 실행 가능 (DB 리셋 가능한 환경)
- 공통 가드레일(반복 액션 차단, 토큰 상한)은 그대로 적용

## 토큰 상한 (1주차 baseline)

### 모델 정책
1주차는 **모든 단계에서 Claude Opus 4.7 단일 모델**을 사용한다.
이유: 정직한 baseline 확보. 2주차에 모델 분리(Opus/Sonnet)를 도입했을 때
"모델 분리만으로 토큰 X% 절감" 같은 명확한 비교 데이터를 얻기 위함.

### 단계별 토큰 상한
- Stage 1 (탐색/매핑): **50k tokens** (input + output 합산)
- Stage 2 (우선순위 판단): **15k tokens**
- Stage 3 (테스트 작성 + 자가 수정 3회): **100k tokens**
- **총 상한: 165k tokens / 1회 실행**

상한 도달 시 단계를 강제 종료하고, 그 시점까지의 결과로 다음 단계 진행.
이는 의도적으로 보수적인 값이며, 상한에 부딪혀 멈추는 것 자체가
2주차 설계의 근거 데이터가 된다 ("어느 단계가 상한을 못 지키는가?").

### Claude Max 플랜 보호
1주차 1회 실행 ≈ 165k 토큰 (Opus 단일). Max 플랜의 일일/주간 한도를
고려해 하루 최대 2~3회 실행을 목표로 한다. 그 이상 필요하면 실험 설계
자체를 재검토해라.

## 1주차 실험 단계
1. 단일 에이전트(Opus)에게 로그인 후 앱 탐색 권한만 주고, 기능 매핑을 스스로
   하게 함 → 결과물: "앱 구조 문서" (마크다운). 일반 사용자 영역 + 관리자
   영역 모두 포함.
2. 매핑된 기능 중 가장 중요해 보이는 플로우 1개를 에이전트가 선택
   → 사람 판단과 비교 (단, 웹소켓 의존 기능은 후보에서 제외)
3. 해당 플로우에 대한 Playwright 테스트 작성 → 실행 → 실패 시 자가 수정 (최대 3회)

## 측정 지표 (주간 리포트용으로 로깅 필요)
### 기본 지표
- 단계별 토큰 소모량 (input/output 분리, cache hit 분리)
- 에이전트가 발견한 기능 수 (일반/관리자 영역 분리해서 집계)
- 테스트 통과까지의 자가 수정 횟수
- 각 단계 소요 시간
- 에이전트의 의사결정 로그 (왜 이 플로우를 골랐는지 등)

### 2주차 멀티 에이전트 + 모델 분리 설계용 추가 지표
- 단계별 토큰 비중 (어디가 가장 무거운가?)
- 단계별 추론 난이도 평가
  (이 단계는 진짜 Opus가 필요한가, Sonnet도 가능한가?)
- 단일 프롬프트 안에서 역할이 섞여 품질이 낮아진 지점
  (예: 우선순위 판단과 코드 작성을 한 번에 시켜서 발생한 문제)
- 자가 수정이 실패한 케이스의 원인 분류
  (탐색 부족 / 코드 버그 / 시나리오 자체 오류 등)

이 지표들이 2주차에 (1) 어떤 에이전트로 분리할지, (2) 각 에이전트에 어떤
모델을 할당할지 결정하는 근거가 된다.

## 작업 요청
다음 순서로 도와줘.

### Step 1: 프로젝트 셋업
- 새 디렉토리 구조 제안 (agent runner / test output / logs / reports 분리)
- 2주차 멀티 에이전트 도입을 고려한 확장 가능한 구조로 설계
  (예: 단계별 핸들러를 모듈화해서 나중에 에이전트별로 분리 가능하게)
- 환경별 가드레일 정책을 설정 파일로 분리 (env.local.ts, env.staging.ts 등)
- 모델 설정도 설정 파일로 분리 (1주차는 모두 Opus 4.7, 2주차에 단계별 분리
  가능하게)
- package.json, tsconfig, 필요한 의존성 정리
- .env.example 작성 (TARGET_URL, LOGIN_URL, TEST_EMAIL, TEST_PASSWORD,
  ANTHROPIC_API_KEY, ENV 등)
- Playwright MCP 연결 설정

### Step 2: 에이전트 코어 작성
- Claude Agent SDK 기반 단일 에이전트 엔트리포인트 (Opus 4.7)
- 1주차 실험 단계 1~3을 순차 실행하는 오케스트레이터
- ENV에 따라 가드레일 정책을 시스템 프롬프트에 주입하는 로직
- 각 단계의 입출력을 파일로 저장 (재현성 + 2주차 비교용)
- 토큰 사용량 로깅 (Anthropic 응답의 usage 필드 활용,
  cache_read_input_tokens / cache_creation_input_tokens 분리 캡처)
- 단계당 토큰 상한 가드 구현 (50k/15k/100k)
- 상한 도달 시 강제 종료 + 그 시점 데이터로 다음 단계 진행

### Step 3: 프롬프트 설계
- 단계 1용: 탐색 + 기능 매핑 프롬프트 (웹소켓 의존 기능은 기록만,
  관리자 영역도 환경 허용 시 탐색)
- 단계 2용: 우선순위 판단 프롬프트 (웹소켓 의존 기능은 후보 제외)
- 단계 3용: 테스트 작성 + 자가 수정 프롬프트 (환경별 가드레일 준수)
- 각 프롬프트는 별도 파일로 분리해서 버전 관리 가능하게
  (2주차에 그대로 각 에이전트 시스템 프롬프트로 재활용 가능하게 설계)

### Step 4: 리포트 자동 생성
- 실행 종료 시 마크다운 리포트 자동 생성
- 포함 내용:
  - 기본 지표 (토큰, 시간, 발견 기능 수 등)
  - 2주차 설계용 추가 지표 (단계별 토큰 비중, 추론 난이도 평가, 역할 섞임 문제)
  - 토큰 상한 도달 여부 및 어느 단계에서 도달했는지
  - 에이전트 결정 로그
  - 생성된 테스트 코드 링크
- 리포트 끝에 "2주차 설계 제안" 섹션 자동 생성
  - 어느 역할을 분리할지 제안
  - 분리된 각 역할에 어떤 모델(Opus vs Sonnet)을 할당할지 제안
  - 1주차 데이터 기반 근거 명시

## 진행 방식
- 각 Step 시작 전에 너의 설계 의도를 짧게 설명하고 내 확인을 받아라.
- 단계마다 동작 확인 후 다음으로 넘어가라. 한 번에 다 짜지 마라.
- 모르는 정보(내 웹앱의 구체 구조 등)는 추측하지 말고 나에게 물어봐라.
- 환경변수, API 키 같은 민감 정보는 절대 코드에 하드코딩하지 마라.
- 2주차 멀티 에이전트 + 모델 분리 확장을 염두에 두되, 1주차에는 단일 에이전트
  + Opus 단일 모델 baseline에 집중한다. 미리 멀티/분리로 만들지 마라.

## 먼저 나에게 물어볼 것
이 프로젝트를 시작하기 전에 네가 모르면 안 되는 정보가 있다면 먼저 질문해라.
예: 로컬에서 웹앱이 실행되는 포트, 로그인 URL 패턴, 관리자 페이지 진입 경로,
테스트하고 싶은 핵심 기능의 대략적 성격, 테스트 계정 및 시드 데이터 준비 여부
등. 추측해서 진행하지 말 것.

자, Step 1부터 시작하자.
